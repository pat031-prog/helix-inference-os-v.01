1. Executive Summary

Helix hoy es un stack local de inferencia y agentes con:
- runtime propio para modelos exportados a tensores locales,
- backend GGUF via llama-cpp-python,
- workspace de modelos con alias,
- asistentes base (general, code, legal),
- benchmark suite para calidad local y tool calling,
- API con streaming SSE,
- frontend local de demo sin build system,
- investigación propia de compresión de KV cache.

La conclusión técnica principal de esta etapa es:
- el modelo base local Qwen3.5-4B Q4 quedó integrado y usable,
- tool calling pasó de 0% a 75% corrigiendo parser + template + calibración,
- la mejor compresión real de KV cache hoy es nuestra línea TurboQuant propia,
- Hadamard ya es el default para turbo-int8,
- turbo-4bit ya es funcional,
- turbo-qjl mejoró fuerte pero sigue en desarrollo.

8.4 Hot-Window Exacto + Pasado Comprimido

Implementación:
- se agregó un contenedor de cache con:
  - cold prefix comprimido
  - hot tail exacto en float32
- el hot-window se cableó a runtime, CLI, benchmarks y sesiones
- flag nuevo: --kv-hot-window

Objetivo:
- mantener exactos los últimos N tokens
- comprimir solo el pasado lejano
- mejorar fidelidad en la parte más sensible de la atención
- preparar el camino para speedups futuros con atención selectiva

Comparado contra no-hot:
- hot-window mejoró bastante el error de logits de 4bit/qjl en GPT2 exportado
- la secuencia todavía no matchea baseline en ese caso
- el tiempo quedó casi igual; todavía no hay speedup real

Conclusión:
- hot-window ya sirve como mejora de fidelidad y como base arquitectónica.
- todavía no da aceleración porque seguimos materializando el pasado frío completo.
- el siguiente paso de research/productividad real es:
  - score aproximado sobre cold prefix
  - exact decode solo para top-K relevantes
  - hot window exacto para el tramo reciente
